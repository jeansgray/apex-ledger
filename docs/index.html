Three-service architecture

Apex Ledger is not a single web app. The council UI on :8080 calls two HTTP backends:

ServicePortWhat it does
Apex Ledger 8080 Dashboard UI, SQLite ledger, council pipeline, human approval gates
Kronos API 5002 Quant price forecasts per holding (animated charts in the UI rail)
MiroFish 5001 Narrative scenario simulation — investor reactions, timelines, reports (AGPL submodule)

Council merges both signals into suggested moves. Missing Kronos → forecasts fall back to fixtures; missing MiroFish → demo simulation fixture only.

Your portfolio (not demo data)

First launch seeds demo holdings — VTI, AAPL, BND plus sample bank charges in ./data/ledger.db. Forecasts and suggested moves target whatever is in that file until you replace it.

Check mode:

curl -s http://127.0.0.1:8080/config | python3 -m json.tool
# "ledger_mode": "demo" or "personal"

Import your holdings from CSV:

cp examples/holdings.example.csv my_holdings.csv
# columns: symbol, quantity, cost_basis, account
python3 scripts/import_personal_ledger.py my_holdings.csv

Restart Apex, open http://127.0.0.1:8080, click Analyze. Status pill shows Your portfolio. Full details: SETUP.md §9.

Not built yet: broker OAuth, bank sync, or a settings screen — CSV import is the supported path today.

Clone & first-time setup

Always initialize the MiroFish submodule (AGPL, HTTP-only boundary):

git clone --recurse-submodules https://gitlab.com/jeansgray/apex-ledger.git
cd apex-ledger
cp .env.example .env
# Edit .env — never commit secrets
python3 scripts/configure_keys.py   # syncs keys into MiroFish env files
uv sync --extra dev
python3 scripts/setup_mirofish.py --no-start

GitHub mirror:

git clone --recurse-submodules https://github.com/jeansgray/apex-ledger.git

Full checklist: SETUP.md in the repo.

Demo vs live API keys

Demo — no keys required Live — LLM + Zep keys
Mode Keys Behavior
Demo None Uses seeded fixture sim_apex_personal_investor (read-only MiroFish APIs)
Live simulations LLM_API_KEY, ZEP_API_KEY in .env Background MiroFish runs (often 5–15 min). Get keys from OpenAI and Zep Cloud.

After adding keys, run python3 scripts/configure_keys.py and restart both services.

Run locally (three terminals)

All three services are required for the full dashboard (forecasts rail, council debate, suggested moves).

Terminal 1 — Kronos (port 5002)

python3 scripts/run_kronos_api.py

Terminal 2 — MiroFish (port 5001)

uv python install 3.12
UV_PYTHON=$(uv python find 3.12) uv run --directory vendor/mirofish/backend python run.py

Terminal 3 — Apex Ledger (port 8080)

uv run apex-ledger serve --port 8080

Open http://127.0.0.1:8080 · Verify: curl -s localhost:8080/health should show "kronos":"ok" and "mirofish":"ok".

Docker Compose

From the repo root (requires submodule + .env):

docker compose up --build

Apex UI: http://127.0.0.1:8080 · MiroFish: :5001 · Kronos forecasts: :5002

Install Docker Desktop (macOS): Docker install guide.

Service boundaries

vendor/mirofish is AGPL-3.0. Do not import MiroFish Python code into src/apex_ledger/. Call it over HTTP via MIROFISH_BASE_URL.

Kronos runs as a separate MIT-licensed service at KRONOS_BASE_URL (default http://127.0.0.1:5002). Same HTTP-only rule — no imports from services/kronos_api into council code beyond the existing client.

Apex Ledger — proprietary app + AGPL MiroFish service boundary. Documentation site only; interactive UI is local.
